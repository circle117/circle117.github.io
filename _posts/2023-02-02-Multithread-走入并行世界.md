## 1.2 概念

方法调用

* **同步Synchronous**：等到方法调用返回后，继续后续的行为
* **异步Asynchronous**：立即返回，在另一个线程中执行

并发与并行

* **并发Concurrency**：多个任务交替执行，多个任务之间可能还是串行（一会儿执行A，一会儿执行B）
* **并行Parallelism**：同时执行（只可能出现在有多个CPU的系统中）

**临界区**：一种公共资源/共享数据，可以被多个线程使用；同时只能有一个线程使用它，一旦临界区资源被占用，其他线程想要使用这个资源就必须等待

多线程间的相互影响

* **阻塞Blocking**：临界区资源被占用，需要该资源的其他线程挂起等待
* **非阻塞Non-Blocking**：没有一个线程可以妨碍其他线程执行

多线程的活跃性问题

* **死锁Deadlock**
* **饥饿Starvation**：某一个或多个线程因为种种原因无法获得所需要的资源，导致一直无法执行
  * 线程优先级低
  * 某一个线程一直占用关键资源，其他需要这个资源的线程无法正常执行
* **活锁Livelock**：资源不断在两个线程中跳动，没有一个线程可以同时拿到所有资源而正常执行

## 1.3 并发级别

**阻塞Blocking**：在其他线程释放资源之前，当前线程被挂起等待

* synchronized & 重入锁
* 悲观策略

**无饥饿Starvation-Free**

* 对于非公平的锁，系统允许高优先级的线程插队，可能导致低优先级线程产生饥饿
* 对于公平的锁，不会产生饥饿

**无障碍Obstruction-Free**：一旦检测到共享数据改坏了，立即对所做的修改进行回滚

* 乐观 策略
* 依赖<u>一致性标记</u>
  * 线程在操作之前，先读取并保存这个标记；在操作完成后，再次读取检查这个标记是否被更改过
    * 一致：资源访问没有冲突
    * 不一致：与其他写线程冲突，重试
  * 任何对资源有修改操作的线程，在修改数据前，需要更新这个一致性标记

**无锁Lock-Free**：保证必然有一个线程能够在有限步内完成操作离开临界区

* 是无障碍的

**无等待Wait-Free**：要求所有的线程都必须在有限步内完成

* 典型结构：RCU (Read-Copy-Update)
  * 读数据：不加控制
  * 写数据：先取得原始数据的副本，只修改副本数据，修改完成后在合适的时机写数据

## 1.4 关于并行的两个定律

### Amdahl定律

定义了串行系统并行化后的<u>加速比</u>的计算公式和理论上限
$$
加速比=优化前系统耗时/优化后系统耗时\\
T_n = T_1(F+\frac{1}{n}(1-F))\\
加速比=\frac{T_1}{T_n}=\frac{T_1}{T_1(F+\frac{1}{n}(1-F))}=\frac{1}{F+\frac{1}{n}(1-F)}
$$

* 加速比越高，优化效果越明显

* 程序中只能<u>串行执行的比例</u>F，<u>处理器个数</u>n

* 为了提高系统的速度，仅增加CPU处理器的数量不一定能起到有效的作用。

  需要从根本上修改程序的串行行为，提高系统内可并行化的模块比重，在此基础上，合理增加并行处理器数量。

### Gustafson定律

$$
执行时间：a+b\\
总执行时间：a+nb\\
加速比：\frac{a+nb}{a+b}\\
定义：F=\frac{a}{a+b}\\
加速比=\frac{a}{a+b}+\frac{nb}{a+b}=F+n(1-F)=n-F(n-1)
$$

* 串行执行时间a，并行执行时间b，处理器个数n

* 如果串行化比例很小，并行化比例很大，加速比就是处理器的个数

### Amdahl定律和Gustafson定律是否相互矛盾

两者的差异是他们的偏重点不同

Amdahl强调：当串行比例一定时，加速比是有上限的，不管堆叠多少个CPU参与计算，都不能突破上限

Gustafson定律强调：如果可被并行化的代码所占比重足够多，那么加速比就能随着CPU的数量线性增长

## 1.5 Java Memory Model

### 原子性Atomicity

一个操作是不可中断的

* 给静态全局变量i赋值
* 对于32位系统来说，long数据的读写不是原子性的（long有64位）

### 可见性Visibility

当一个线程修改了某一个共享变量的值，其他线程是否能够立即知道这个修改

* 缓存优化
* 硬件优化
* 指令重排：修改线程中指令执行顺序
* 编辑器的优化：向前替换

### 有序性Ordering

指令重排保证串行语义的一致性，不会使串行的语义逻辑发生问题

```java
class OrderExample {
  int a=0;
  boolean flag = false;
  public void writer() {
    a = 1;
    flag = true;
  }
  public void reader() {
    if (flag) {
      int i = a+1;
    }
  }
}
```

* 线程A执行writer()方法，线程B执行reader()方法
* 发生指令重排，不能保证flag=true时，a被赋值为1

指令重排原因

* 指令执行分为以下几步
  1. 取指IF
  2. 译码和取寄存器操作数ID
  3. 执行或者有效地址计算EX
  4. 存储器访问MEM
  5. 写会WB
* 流水线技术执行指令
* 通过指令重排，减少流水线中断

### 哪些指令不能重排：Happen-Before原则

<u>程序顺序原则</u>：一个线程内保证语义的串行性

<u>volatile原则</u>：volatile变量的写，先发生于读，保证了volatile变量的可见行

<u>锁规则</u>：解锁必然发生在随后的加锁前

<u>传递性</u>：A先于B，B先于C，那么A先于C

线程的start()方法先于它的每一个动作

线程的所有操作先于<u>线程的终结</u>（Thread.join()）

线程的<u>中断</u>（interrupt()）先于被中断线程的代码

对象的<u>构造函数</u>执行、结束先于finalie()方法
